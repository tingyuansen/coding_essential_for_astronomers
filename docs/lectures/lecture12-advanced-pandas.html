Advanced Pandas Operations for Astronomical Catalogs¶

Introduction¶

In the previous lecture, you learned the fundamentals of working with Pandas DataFrames: loading catalogs, selecting columns and rows, filtering with boolean conditions, adding calculated columns, and handling missing data. These skills let you explore and clean individual astronomical catalogs.

But real astronomical research is rarely that simple. You don't just work with one catalog in isolation. Instead, you need to compare across groups - how do nearby red dwarfs differ from distant ones? Are high-velocity stars systematically brighter or fainter? Questions like these require splitting your catalog into meaningful groups and computing statistics for each.

You also need to combine multiple catalogs. Your Gaia positions need to be matched with spectroscopic observations from last night. Photometry from SDSS needs to be cross-referenced with proper motions from Gaia. Much of astronomical research involves merging data from different sources. Three nights of spectroscopy produce three separate files. Multiple sky regions need to be combined into one catalog. You need to vertically stack DataFrames while preserving data integrity.

Multi-dimensional grouping creates complex indices that require special navigation techniques. Understanding how to work with and reshape these hierarchical structures is essential for advanced analysis. You'll need to create tables showing relationships like "for each spectral type and distance range, what's the average radial velocity?" These cross-tabulated summaries reveal patterns that simple statistics miss.

Sometimes you need more than magnitude - 5 * np.log10(distance). You need conditional logic or complex transformations that require custom functions. You need to calculate smoothed profiles along distance, find cumulative distributions, and identify local outliers. Window functions let you compute statistics over sliding or expanding intervals.

Today's lecture introduces seven advanced Pandas operations that professional astronomers use daily.

Adding Derived Quantities¶

From Lecture 11, we learned to create new columns using vectorized operations - calculations that apply to entire columns simultaneously. Let's recreate the key derived quantities we'll use throughout today's analysis.

The relationship distance = 1000 / parallax converts Gaia's parallax measurements in milliarcseconds to distances in parsecs. The distance modulus formula M = m - 5*log₁₀(d/10) converts apparent magnitude, which is how bright a star appears, to absolute magnitude, its intrinsic brightness. The difference BP - RP between blue and red photometry tells us stellar temperature - small values mean hot blue stars, while large values mean cool red stars.

# Add derived columns using vectorized operations

GroupBy: The Split-Apply-Combine Strategy¶

GroupBy is one of the most powerful operations in Pandas, essential for answering comparative questions in astronomy.

The Problem GroupBy Solves¶

Imagine you want to answer: "How does average luminosity differ between hot blue stars and cool red stars?" With Lecture 11 techniques, you'd write:

blue_stars = gaia[gaia['bp_rp'] < 0.5]
red_stars = gaia[gaia['bp_rp'] > 2.0]
blue_mean = blue_stars['abs_g_mag'].mean()
red_mean = red_stars['abs_g_mag'].mean()

This works for two groups, but becomes tedious for five color categories or ten distance bins. GroupBy solves this elegantly through split-apply-combine. First, you split the DataFrame into groups, perhaps by color category. Then you apply calculations to each group, such as mean magnitude. Finally, you combine results into a summary.

Creating Categories and Basic Grouping¶

Before grouping by color, we need to convert our continuous bp_rp values into discrete categories. We use pd.cut(), introduced in Lecture 11, which divides continuous data into labeled bins. The bins parameter specifies the boundaries, and labels provides human-readable names for each category.

# Create color categories
gaia['color_category'] = pd.cut(
    gaia['bp_rp'],
    bins=[-1, 0.5, 1.5, 2.5, 5],
    labels=['Blue', 'Yellow-White', 'Orange', 'Red']
)

# Group by color and calculate mean absolute magnitude
color_groups = gaia.groupby('color_category', observed=False)
mean_mag_by_color = color_groups['abs_g_mag'].mean()

print("Mean absolute magnitude by color:")
print(mean_mag_by_color)

The observed=False parameter is important for categorical data - it tells Pandas to include all categories we defined (Blue, Yellow-White, Orange, Red) even if some have zero stars. This prevents future warnings and ensures consistent output structure.

Red stars have higher, and therefore fainter, absolute magnitudes than blue stars - they're intrinsically less luminous. This makes physical sense because red stars are cooler and smaller, radiating less energy per unit area and having less total surface area. The progression from blue to red traces the main sequence from hot massive stars to cool dwarfs.

Understanding .size() vs .count()¶

When counting objects in groups, Pandas offers two methods with an important distinction. The .size() method counts ALL objects in each group, including those with NaN values in any column. It returns the total number of rows per group regardless of missing data. On the other hand, .count() counts only NON-NULL values in a specific column. If you call it on a grouped column, it tells you how many valid measurements exist.

Multiple Statistics with .agg()¶

Usually you want several statistics simultaneously, not just the mean. The .agg() method, short for "aggregate", lets you compute multiple functions at once by passing a list of function names. Common aggregation functions include 'count' for the number of non-null values in each group, 'mean' and 'median' for central tendency measures, 'std' and 'var' for spread measures (standard deviation and variance), 'min' and 'max' for range endpoints, 'sum' for the total of all values, and 'first' or 'last' for the first or last value in each group.

# Multiple statistics for absolute magnitude
color_stats = color_groups['abs_g_mag'].agg(['count', 'mean', 'std', 'min', 'max'])

Grouping by Multiple Columns¶

The real power of GroupBy emerges when you group by multiple columns simultaneously, creating fine-grained subpopulations. Let's create distance bins and examine how stellar populations vary across both distance and color. We'll use .size() to count the number of stars in each group - the total number of rows regardless of any missing data in specific columns.

Working with MultiIndex¶

MultiIndex, or hierarchical indexing, appears whenever you group by multiple columns or create pivot tables. Understanding how to navigate, select from, and reshape MultiIndex DataFrames is essential for advanced analysis.

Understanding MultiIndex Structure¶

Let's examine the MultiIndex we just created more closely. A MultiIndex has named levels, like columns in a regular DataFrame, and you can query its structure using several methods:

print("\nFirst level values:", counts_2d.index.get_level_values(0).unique())
print("\nSecond level values:", counts_2d.index.get_level_values(1).unique())

The MultiIndex has two levels: distance_bin at level 0 and color_category at level 1. The .names attribute shows what each level is called. The .get_level_values() method extracts all values for a specific level - here we see all four distance bins and all four color categories. Each combination of these values represents a unique row in our Series.

Selecting from MultiIndex with .xs()¶

The .xs() method, which stands for "cross-section", selects data at a particular level of the MultiIndex. This is much cleaner than complex boolean indexing when working with hierarchical data. You specify the value you want to select and which level it belongs to.

# Get all data for the 50-75 pc distance bin
mid_distance = counts_2d.xs('50-75 pc', level='distance_bin')
print("Star counts in 50-75 pc shell by color:")
print(mid_distance)

# Get all red stars across all distances
red_stars = counts_2d.xs('Red', level='color_category')
print("\nRed star counts by distance:")
print(red_stars)

The level parameter specifies which index level to slice. You can use either the level name, such as 'distance_bin', or its position, 0 or 1. The .xs() method returns a Series with the remaining index levels - when you slice on distance_bin, you get back a Series indexed by color_category, and vice versa.

This makes it easy to answer questions like "How many stars of each color are in the 50-75 pc shell?" or "How does the red star count vary with distance?"

Reshaping with .unstack()¶

The .unstack() method pivots one index level into columns, converting a MultiIndex Series into a regular DataFrame. This transformation is called "unstacking" because you're taking a "stacked" long-format structure and spreading it out into a "wide" format. The result is much more readable for humans and perfect for creating tables.

# Convert MultiIndex Series to DataFrame with colors as columns
counts_wide = counts_2d.unstack(level='color_category', fill_value=0)
print("Star counts reshaped (distance × color):")
print(counts_wide)

This wide format is exactly what you'd create by hand if making a table for a paper: rows are one variable, distance, columns are another, color, and cells contain the data, star counts.

Reshaping with .stack()¶

The .stack() method does the opposite transformation - it pivots column labels into an index level, converting a wide DataFrame back to a long MultiIndex format. This is the inverse operation of .unstack().

Use .unstack() to make data more readable for humans in tables and reports, and .stack() to prepare data for operations that expect long format, such as certain plotting functions and statistical analyses.

When to Use MultiIndex vs Avoid It¶

MultiIndex is powerful but not always the right choice.

Use MultiIndex when you're doing intermediate calculations on grouped data, you need to efficiently select specific subgroups with .xs(), your operations naturally work on hierarchical data, or you're manipulating data before final presentation.

Flatten MultiIndex when creating tables for publication or reports using .unstack(), passing data to plotting functions that expect simple indices, exporting to CSV since MultiIndex doesn't export cleanly, or sharing data with collaborators who may not understand hierarchical indexing.

Pivot Tables: A Convenient Alternative to GroupBy + Unstack¶

We've seen how to create cross-tabulated summaries using .groupby() followed by .unstack(). Pivot tables provide a more convenient way to achieve the same result in a single operation. The pd.pivot_table() function combines grouping, aggregation, and reshaping into one step.

Basic Pivot Table: Counting Stars¶

Let's create a pivot table to count stars by distance and color category. This will produce the same grid we created earlier with .groupby().size().unstack(), but more directly. We want rows to be distance bins like <25 pc and 25-50 pc, columns to be color categories like Blue and Yellow-White, and cells to contain the number of stars in that distance-color combination.

The pd.pivot_table() function requires several key parameters. The values parameter specifies the column to aggregate - we use 'source_id' since every star has one, making it reliable for counting, and each source_id is unique. The index parameter sets the variable for rows, here distance bins. The columns parameter sets the variable for columns, here color categories. The aggfunc parameter defines the aggregation function - 'count' counts how many non-null values appear in each cell. Finally, fill_value determines what to use for empty cells, 0 instead of NaN.

# Count stars by distance and color
pivot_counts = pd.pivot_table(
    gaia,
    values='source_id',
    index='distance_bin',
    columns='color_category',
    aggfunc='count',
    fill_value=0,
    observed=False
)

This makes astrophysical sense because red dwarfs are the most common stars in the galaxy but so intrinsically faint we can only detect them nearby.

Notice how much easier this is to read than the long MultiIndex Series from our GroupBy! The grid format makes comparisons obvious - you can scan down columns to see how each color's count changes with distance, or across rows to see each shell's color distribution.

Visualizing Pivot Tables with Heatmaps¶

Pivot tables are naturally suited to heatmap visualization, where cell color indicates value magnitude. This makes patterns pop visually that might be subtle in the numbers alone. We'll use matplotlib's imshow() to create a color-coded grid where darker colors represent higher star counts.

The heatmap makes population distributions visually obvious. The "Red" column is darker, indicating more stars, across all distance bins, especially in the nearest bin. The intensity gradient from red, high counts, to light yellow, low counts, immediately shows where stars concentrate.

This visualization reveals patterns harder to see in raw numbers. Red stars dominate everywhere, blue stars are rare nearby but more common far away suggesting detection bias, and the sample size grows with distance because spherical shells have larger volumes.

Merging: Cross-Matching Multiple Catalogs¶

Most astronomical research requires combining data from different sources - Gaia astrometry with ground-based spectroscopy, photometry from SDSS with proper motions from Gaia, X-ray detections with optical counterparts. Merging, also called "joining" or "cross-matching", combines DataFrames by matching rows that share a common identifier.

Creating a Realistic Observational Log¶

Let's simulate a realistic scenario: you've spent several nights observing a subset of our Gaia stars. Your observational log records which stars you observed and under what conditions - seeing quality, telescope used, exposure times, and your post-observation quality assessment. This metadata is crucial for determining whether your observations are scientifically useful.

Inner Join: Successfully Observed Stars Only¶

Let's start with an inner join - the most conservative approach that only keeps stars appearing in both catalogs.

The pd.merge() function is Pandas' workhorse for combining DataFrames. Its first argument is the "left" DataFrame, here gaia. The second argument is the "right" DataFrame, here obs_log. The on parameter specifies the column names to match on, here 'source_id', which must exist in both DataFrames. The how parameter sets the merge type: 'inner', 'left', 'right', or 'outer'. The optional suffixes parameter handles overlapping column names, defaulting to '_x' and '_y'.

# Inner join: only stars with both Gaia data AND observations
observed_stars = pd.merge(
    gaia,
    obs_log,
    on='source_id',
    how='inner'
)

print(f"Result: {len(observed_stars)} rows (observed stars only)")
observed_stars[['source_id', 'phot_g_mean_mag', 'abs_g_mag', 'obs_date', 'seeing']].head()

Merging with Different Column Names¶

Real catalogs often use different names for the same identifier. For example, Gaia might use 'source_id' while your catalog uses 'gaia_id'. The left_on and right_on parameters handle this:

# Simulate a catalog with different column name
obs_log_renamed = obs_log.rename(columns={'source_id': 'gaia_id'})

# Merge using left_on and right_on

Concatenating: Stacking Multiple DataFrames¶

While merging combines DataFrames horizontally by adding columns through matching keys, concatenating stacks them vertically by adding rows. This is essential when combining multiple observation runs into a single catalog, stitching together different sky regions observed separately, appending new observations to an existing catalog, or assembling catalog chunks downloaded in pieces.

Understanding Index Behavior Without ignore_index¶

Without ignore_index=True, pandas preserves the original index from each DataFrame. This means you can have duplicate index values in the result!

If each night's DataFrame had rows indexed 0-49, 0-59, and 0-39, the concatenated result will have multiple rows with the same index value, for example three different rows all with index 0. This isn't an error - it's the intended behavior when you don't use ignore_index.

This can cause problems. Using .loc[0] returns multiple rows instead of one. Some operations expect unique indices. It's confusing and hard to debug.

Concatenate vs Merge: When to Use Each¶

People often confuse concatenating and merging. Here's the key distinction.

Use concatenate with pd.concat() when stacking data from multiple observation runs with similar structure, combining catalog chunks like different sky regions or different quarters, appending new observations to existing catalogs, or you want to stack rows vertically to make your DataFrame longer. In these cases, the rows represent different objects or different observations.

Use merge with pd.merge() when cross-matching catalogs from different sources like Gaia + SDSS or Gaia + your observations, adding supplementary data to stars by matching IDs, or you want to combine data horizontally by matching keys to make your DataFrame wider. Here, the rows represent the same objects with different information.

Think of it this way: concatenate is like stacking papers on top of each other. Merge is like matching information from two ledgers using an ID number.

Apply: Custom Functions Beyond Vectorized Operations¶

In Lecture 11, we created new columns using vectorized operations - simple arithmetic applied to entire arrays at once: gaia['distance_pc'] = 1000 / gaia['parallax']. These are blazingly fast because NumPy performs calculations in compiled C code on entire arrays simultaneously.

But some calculations can't be vectorized. They require conditional logic, like "If magnitude < 10 classify as 'bright', else if < 15 'medium', else 'faint'". They need row-wise operations, calculations requiring multiple column values examined together per row. Or they involve complex transformations like string parsing, date manipulations, or custom algorithms.

The .apply() method lets you run custom Python functions on DataFrame columns or rows. It's slower, often 10 to 100 times compared to vectorized operations, but infinitely more flexible.

When to Use Apply vs Vectorized Operations¶

The .apply() method loops through rows in Python, calling your function each time. Vectorized operations process entire arrays in C. For 5000 rows, .apply() might take longer to compute than vectorization. But sometimes you have no choice - complex logic simply can't be vectorized.

Apply with Conditional Logic: Stellar Classification¶

Let's create a stellar classification using multiple properties. We'll examine each star's position on the Hertzsprung-Russell diagram, which plots absolute magnitude versus color, and assign it to a stellar class. This requires nested conditional logic - checking multiple thresholds in different combinations for different stellar types.

def classify_star(row):
    """
    Classify stars using simplified HR diagram logic.
    Takes a DataFrame row and returns a classification string.
    """
    abs_mag = row['abs_g_mag']
    color = row['bp_rp']
    
    # Giants: bright (low abs mag) and red (high color index)
    if abs_mag < 3 and color > 1.5:
        return 'Giant'
    # White dwarfs: faint (high abs mag) and blue (low color index)
    elif abs_mag > 10 and color < 0.5:
        return 'White Dwarf'
    # Main sequence: classify by color
    elif 3 <= abs_mag <= 15:
        if color < 0.5:
            return 'MS: Hot (OBA)'
        elif color < 1.5:
            return 'MS: Sun-like (FGK)'
        else:
            return 'MS: Cool (M)'
    else:
        return 'Unclassified'

# Apply classification to each row (axis=1 means row-wise)
gaia['stellar_class'] = gaia.apply(classify_star, axis=1)

The classify_star function receives an entire DataFrame row as input because we specified axis=1. Inside the function, row['abs_g_mag'] extracts the absolute magnitude from that specific star's row, row['bp_rp'] gets that star's color index, and the nested if-elif structure implements HR diagram decision boundaries.

The axis=1 parameter is crucial - it tells .apply() to pass each row to the function. The axis=1 means "along the column axis", in other words across columns within a row. If you used axis=0, it would pass each column instead, which wouldn't make sense here.

Why can't we vectorize this? The logic has different paths with different criteria. Giants need BOTH abs_mag < 3 AND color > 1.5. White dwarfs need abs_mag > 10 AND color < 0.5. Main sequence stars need magnitude in range, THEN sub-classify by color.

You could build this with boolean masking and multiple .loc[] assignments, but it would be much less readable and error-prone. The .apply() method trades speed for clarity.

Apply for Multi-Column Calculations: 3D Velocities¶

Let's calculate total 3D space velocities from proper motions, which measure motion across the sky, and radial velocities, which measure motion toward or away from us. This requires combining two proper motion components in RA and Dec, converting to tangential velocity using distance, combining with radial velocity using the Pythagorean theorem, and handling missing radial velocities gracefully.

This multi-step calculation with conditional logic and multiple columns needs .apply().

def calculate_total_velocity(row):
    """
    Calculate 3D space velocity from proper motions and radial velocity.
    Returns NaN if radial velocity is missing.
    """
    # Check if we have radial velocity - if not, can't compute 3D velocity
    if pd.isna(row['radial_velocity']):
        return np.nan
    
    # Combine RA and Dec proper motions to get total proper motion (mas/yr)
    pm_total = np.sqrt(row['pmra']**2 + row['pmdec']**2)
    
    # Convert to tangential velocity in km/s
    # Formula: v_tan = 4.74 * μ (mas/yr) * d (pc)
    # The factor 4.74 comes from unit conversions
    v_tan = 4.74 * pm_total * row['distance_pc']
    
    # Combine tangential and radial velocities (perpendicular components)
    v_total = np.sqrt(v_tan**2 + row['radial_velocity']**2)
    
    return v_total

gaia['v_total'] = gaia.apply(calculate_total_velocity, axis=1)

print(f"Missing (no radial velocity): {gaia['v_total'].isna().sum()} stars")
print(f"\nVelocity statistics:")

Window Functions: Rolling and Expanding Statistics¶

Window functions compute statistics over sliding or expanding intervals of your data. While commonly used for time series like stock prices or climate data, they're equally valuable for spatial analysis in astronomy: smoothing profiles along distance, identifying local outliers, or computing cumulative distributions.

The key concept is that instead of computing statistics for the entire dataset, you compute them over a "window" that moves through your data or grows from the beginning.

Rolling Windows: Moving Averages¶

A rolling window computes statistics over a fixed-size sliding window. Imagine a window that contains N consecutive rows, specified by the window parameter, slides down your DataFrame one row at a time, and computes a statistic like mean, std, min, or max for each position.

The center=True parameter centers the window on each point, using rows before and after, rather than looking only backward. This produces smoother results for spatial data.

By default, if ANY value in the window is NaN, the result is NaN. This is problematic for real data with scattered missing values. The min_periods parameter tells rolling to compute the statistic as long as at least that many non-NaN values exist in the window. Setting min_periods=50 for a window=100 means "compute the mean if you have at least 50 valid values, even if the other 50 are NaN or don't exist at the edges."

Let's sort stars by distance and compute rolling averages to smooth the magnitude-distance relationship:

# Sort by distance for spatial analysis
gaia_sorted = gaia.sort_values('distance_pc').reset_index(drop=True)

# Calculate rolling mean of absolute magnitude (window of 100 stars)
# min_periods=50 means: compute if at least 50 non-NaN values exist
gaia_sorted['abs_mag_rolling'] = gaia_sorted['abs_g_mag'].rolling(
    window=100, center=True, min_periods=50
).mean()

# Calculate rolling mean of color
gaia_sorted['color_rolling'] = gaia_sorted['bp_rp'].rolling(
    window=100, center=True, min_periods=50
).mean()

# Show sample from middle where rolling mean is well-defined
print("Sample from middle of catalog:")
print(gaia_sorted[['distance_pc', 'abs_g_mag', 'abs_mag_rolling', 
                   'bp_rp', 'color_rolling']].iloc[100:110])

Our sample is volume-limited by distance, but within that volume we don't detect all stars - we miss the faintest ones at large distances. The rolling average reveals this bias: at 100 pc, we're systematically detecting brighter stars than exist nearby.

The right panel shows color versus distance. Color shifts systematically bluer, meaning lower BP-RP, with distance. This correlates with the magnitude effect - we preferentially detect hotter, more luminous blue stars far away, while we see the full population including faint red dwarfs close by. The red stars we detect at 100 pc are brighter-than-typical red stars; the typical faint red dwarfs are invisible at that distance.

These rolling statistics quantify selection biases that would be hard to see in the raw scatter. They're essential for understanding your sample's completeness and correcting scientific measurements.

Expanding Windows: Cumulative Statistics¶

An expanding window computes cumulative statistics from the start of the DataFrame up to each row. Unlike a rolling window with fixed size that slides through data, an expanding window starts with just the first row, grows to include more rows as you move down, and at row N, includes all rows from 0 to N.

This is perfect for cumulative analyses: "As I include stars out to distance D, how do population properties change?"

# Calculate cumulative statistics
gaia_sorted['cumulative_mean_mag'] = gaia_sorted['abs_g_mag'].expanding().mean()
gaia_sorted['cumulative_std_mag'] = gaia_sorted['abs_g_mag'].expanding().std()
gaia_sorted['cumulative_count'] = gaia_sorted['abs_g_mag'].expanding().count()

print("Cumulative statistics at different distances:")
print(gaia_sorted[['distance_pc', 'cumulative_count', 'cumulative_mean_mag', 
                   'cumulative_std_mag']].iloc[::500])  # Every 500th row

The left panel shows cumulative mean magnitude. It stabilizes around 40-50 pc as we've sampled the local stellar population, then slowly decreases at larger distances due to Malmquist bias - we miss faint stars far away, pulling the mean brighter. The stabilization point tells us where our sample becomes "complete" - beyond approximately 50 pc, we're no longer seeing the full population.

The right panel shows cumulative standard deviation. It increases rapidly at first as we sample diverse star types, then plateaus around 60-70 pc. Initially the sample is dominated by whatever stars happen to be very close, causing high variance from small samples. Then it stabilizes once we've sampled enough stars to characterize the population. Finally it starts increasing again as selection effects let rare bright stars dominate at large distances.

These cumulative plots answer essential questions: "At what distance does my sample properties stop changing? Where does incompleteness start affecting results?" These are essential questions for any distance-limited survey.
